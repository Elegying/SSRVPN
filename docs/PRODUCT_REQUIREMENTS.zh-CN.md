# SSRVPN 产品行为要求

这份清单用于开发、测试和发布检查。改动打包、首页、订阅、节点列表、测速或连接流程时，必须确认以下行为仍然成立。

## 安装包

1. macOS 发布产物必须是可拖拽安装的 DMG：根目录包含 `SSRVPN.app` 和指向 `/Applications` 的 `Applications` 快捷方式。
2. Windows 必须同时发布每用户安装版 `SSRVPN_Setup.exe` 和解压即用的便携式 ZIP。安装器固定写入 `%LOCALAPPDATA%\Programs\SSRVPN`、不请求管理员权限，升级时应先恢复自有系统代理并自动关闭 SSRVPN 专属进程；随后只替换已知程序文件，保留安装版的订阅、设置、DPAPI 密钥、LocalAppData 回退数据和窗口状态，且不搜索或修改其他位置的便携副本。便携包解压后的主目录只能有一个面向用户的 `.exe`，即 `ssrvpn_windows.exe`。

## 首次使用和订阅

3. 首次打开或检测到没有节点时，自动弹窗要求用户导入订阅链接或节点链接。
4. 用户导入订阅链接或节点链接后，应用自动刷新订阅，主页自动加载节点并开始测速。

## 节点列表

5. 节点测速超时后自动置底、变灰、不可选择，但仍保留编辑入口。
6. 未超时节点保持订阅原始顺序，不按延迟重新排序。
7. 应用记忆上次连接的节点。Android 快捷磁贴一键连接时默认沿用上次节点。

## 状态可信度与恢复

8. 只有核心进程、受认证 API 与目标平台代理/TUN 都确认就绪后，界面才显示“已连接”；仅端口开放或启动命令返回不算成功。
9. “连接中”必须允许取消；取消、重复点击或较早一次启动的迟到结果不能覆盖更新一代状态。
10. 核心异常退出、权限取消、网络设置恢复失败或订阅刷新失败时，首页、Android 通知和快捷磁贴不能继续显示过期的已连接状态。Android 核心意外退出只允许一次受控恢复；恢复中和最终失败必须明确通知，用户主动断开不得触发自动重连。
11. 刷新或导入失败不能破坏最后一份可用订阅。多订阅刷新部分失败时必须返回结构化的成功与失败来源并显示警告，不能把部分成功误报为全量成功或破坏整批旧状态。错误必须说明失败阶段和用户可以执行的下一步，不直接展示原始 URL、secret、token、密码或节点凭据。
12. 更新下载应展示进度并允许取消；校验失败或取消后清理临时文件，不打开未验证产物。

## 跨平台体验

13. 代理、SOCKS 或 API 端口被占用并自动改用临时端口时，必须向用户列出实际变化；下一次未调整时不能继续显示旧提示。
14. 三端对“未配置、正在连接、已连接、正在断开、失败”的含义保持一致；平台权限和安装差异使用各自指南说明。
15. 动态状态不能只靠颜色表达。主要按钮、节点选择、错误提示和进度需要可读语义、合理焦点顺序与键盘/读屏器操作路径。
16. macOS TUN 在连接前明确说明本次管理员授权的用途；取消授权后返回可重试状态，不自动降级到用户未选择的模式。
17. 首页 IPv4 展示只表示当前 IPv4 出口，不得把它描述为 IPv6 不可用；IPv6 能力以生成配置和实际网络测试为准。

## 验收原则

每项用户行为至少有一个自动回归测试或目标平台可复核清单。只验证“函数执行完成”不等于验收，测试应断言用户最终看到的状态、数据保留和系统网络设置。
