# 项目健康状态

最近审查：2026-07-18<br>
审查对象：基于 `v3.4.5` 的 `v3.4.6` 发布候选分支
`refactor/remove-unlock-and-hardening`。终审冻结时尚未推送、创建 tag、发布 Release 或构建
APK/DMG/Windows 安装包；这些步骤必须在受保护 PR 通过后执行。

## 结论

SSRVPN 当前源码、三端边界、恢复模型和自动化门禁整体健康。本轮完成了以下收敛与加固：

- Android、macOS、Windows 删除解锁测试页面、共享检测服务和专属测试，主导航统一只保留
  “首页/主页”与“订阅”；活动源码和当前用户文档已无残留，并由
  [ADR-004](decisions/004-two-page-product-surface.md) 与仓库守卫固定。
- 三端首页的 SSRVPN 标题右侧统一显示来自共享版本常量的小号版本号，包含读屏语义和产品
  表面守卫，版本不会由平台各自写死。
- 所有生产 YAML 解析入口统一执行 20 MiB 输入、64 层嵌套、256 次别名引用和 100,000 个
  集合项预算；高基数块/流集合会在进入 `loadYaml` 前被拒绝，磁盘缓存也会在读入内存前
  先检查大小，并有超限、端到端拒绝与正常解析回归。
- Android 快捷磁贴和粘性重启改用 Android Keystore 加密的原生 API secret 副本，冷启动不再
  依赖 Flutter 进程先写入明文偏好；常驻通知按完整展示状态去重。
- Android 删除两个无人调用或没有原生实现支撑的旧门面，并把 MethodChannel 分发、Mihomo API
  等待与 VPN protect 管道拆成聚焦组件。
- GeoIP 已更新到 2026-07-18 07:14 的最新已验证快照；来源固定到 GitHub Release 资产 ID、
  原始摘要和确定性 gzip 摘要，定时任务只在完整验证后创建更新 PR，不直接修改主分支；只有
  `OPEN` 的既有 PR 才会复用，关闭或已合并的异常状态会使任务失败而不是假绿。
- 以冻结提交为基线完成三轮常规审查和三轮对抗审查，并对每轮发现完成修复、回归与闭环复核；
  随后三次独立 Codex CLI 复核发现的 Android 拒绝启动顺序、Windows 旧 TUN 误认、GeoIP PR
  状态、超长节点名快照编码、高基数 YAML 和节点切换竞态均已逐项修复并补回归；快照改用
  有界 UTF-8 编码并保留旧版读取兼容。当前未保留已知 P0-P2。
- Android 原生启动、快捷磁贴、粘性恢复和停止流程现在共用单向会话协议：启动配置必须被唯一
  声明后才能消费，旧回调只能释放自己的声明；停止期间持续保护运行配置，Bridge 与 TUN 真正
  退出后才清除快照。已准备配置和被原生操作阻塞的清理也分别受租约与结束后重试保护；失效
  恢复请求无法停止较新的活动或启动中会话，被撤销的启动声明会安全终止对应空闲服务请求。
- Android 手动节点选择在“旧请求成功、最新请求失败”时，以当前连接意图和原生会话代际双重
  核验 Mihomo 实际节点，并同步首页、持久设置和快速启动快照，不会继续显示或恢复失败目标。
- Windows 旧 TUN 标记迁移只有在同一接口同时具备 SSRVPN 专属 IPv4、IPv6 和路由证据时才
  提升为稳定 GUID；其他 VPN 的默认或拆分默认路由不会单独成为 SSRVPN 所有权证据。

Windows 安装数据保留与安装器单一分发决策不变：安装器固定使用每用户目录，只替换程序文件，
不搜索或合并桌面、下载目录及其他旧独立副本。因此多个旧数据源不会进入安装事务，保留安装版
订阅和设置不会重新引入旧的多来源安装失败。边界分别由
[ADR-002](decisions/002-windows-installed-data-preservation.md) 和
[ADR-003](decisions/003-windows-installer-only-distribution.md) 固定。

本轮没有修改 HTTP 订阅策略。Apple/Windows 付费签名也仍不属于项目目标；免费桌面分发缺少
受信任发布者身份是需要持续向用户说明的限制。

## 代码整洁度复核

- 活动 Dart 源码中不存在解锁测试的类、服务、路由或文案；三端导航均由守卫精确限制为两项。
- 未发现循环依赖、重复实现扩散、隐藏全局状态链或无人调用的兼容门面。共享领域逻辑集中在
  `ssrvpn_shared`，平台层承载系统集成，调用方向清晰。
- 知识图谱的高认知复杂度主要位于 Windows 子进程按路径识别、代理/安装恢复脚本、Android
  VPN 启动编排和仓库一致性校验。它们有明确职责、超时/回滚语义、体量上限或回归测试，不是
  无边界互相调用的“屎山”，但仍是后续修改时最需要小步拆分的区域。
- `MainActivity` 的 MethodChannel `when` 分发虽然静态复杂度偏高，但只有 26 行且每个分支已
  委托独立处理器；为降低数字而引入动态注册表不会改善可读性。`startCoreWithVpn` 是剩余的
  Android 关键编排热点，保持单向阶段和统一失败清理，后续只在新增行为时继续提取可测试阶段。
- Android 的核心 Dart 服务和原生 VPN Service 分别为 `836/850` 行、`915/915` 行，均受仓库
  门禁约束；本轮把原生会话状态和 Dart 原生桥接职责提取为独立组件，没有用转移代码掩盖复杂度。
- Windows launcher 和恢复脚本依赖 Windows API 与进程所有权语义；在没有对应 Windows 运行时
  验证的本轮中不做纯行数驱动的大改写。
- 六轮审查及三次独立 Codex CLI 复核后的可复现问题均已关闭；这不表示未来代码永久不会产生缺陷。
  Windows 原生恢复器和 PowerShell 5.1 的真实执行仍必须由 Windows 环境验收。

## 下一步门槛

1. 本地分支若准备合并，必须通过受保护 PR 的全部必需 CI，不绕过主分支保护。
2. PR 合并到 `main` 后创建 `v3.4.6` 标签，执行 tag 级 APK、DMG、Windows 安装器构建，并
   复核 GitHub、OSS 与官网固定入口；不发布 Windows 便携包。
3. 以后有 Android 真机时观察快捷磁贴、系统回收后粘性重启、通知断开和长时间连接耗电。
4. 后续改动触达 Windows 或 Android 原生高复杂度区时，先增加/更新目标平台回归，再小步提取
   独立职责，不做没有运行时证据的大规模重写。
5. 用 TalkBack、VoiceOver 和 Narrator 验证首页连接、订阅导入、诊断恢复和动态状态播报。
