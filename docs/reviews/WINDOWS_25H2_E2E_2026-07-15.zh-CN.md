# Windows 11 25H2 实机端到端测试记录

- 测试日期：2026-07-15
- 基线客户端：SSRVPN 3.4.1
- 修复目标：SSRVPN 3.4.2
- 当前实机：Windows 11 Pro 25H2，build 26200.8655，x64
- 测试性质：真实系统代理、真实客户端进程和真实网络连接
- 当前发布结论：**NO-GO**

> 本文已记录当前 3.4.2 测试安装包在 Pro 25H2 宿主机上的结果。独立 guardian 与登录时 recovery-only RunOnce 的源码已完成，但包含全部最终修复的新安装包尚未完成 Pro 25H2 宿主机全矩阵；管理员 UAC/TUN 更新、Windows 11 Home 25H2 虚拟机、真实重启/`TurnOff` 和正式发布资产的客户端内置更新也仍为 `PENDING`。源码测试不能代替最终安装包实机回归，当前结论保持 **NO-GO**。

## 脱敏说明

测试使用了专用 SSR 导入链接和订阅源。凭据只在本机测试期间使用，未写入仓库。本文不包含公网 IP、节点名称、节点域名、服务器地址、订阅地址或令牌、Windows 用户目录、dump 原文或完整原始调用栈。

## 判定标准与基线

- `PASS`：当前测试安装包在普通 Windows 用户上下文中已有完整实机证据。
- `BLOCKED`：当前构建存在已复现缺陷，或缺少目标系统、真实重启、最终修复安装包等发布前置条件；均不得视为通过。

测试开始前确认 WinINet 系统代理关闭、WinHTTP 直连，没有 SSRVPN 核心进程、本地代理监听或待处理恢复记录。正常断开、取消连接和异常退出均以准确恢复该基线为通过条件。

## 测试夹具边界

Computer Use 的 `sky.launch_app` 会让被启动进程看到隔离或虚拟化的 HKCU。该上下文与普通桌面 Shell 属于同一 Windows 用户和会话，但 Internet Settings 的可见值不同。因此，经 `sky.launch_app` 启动安装器时出现的那一次退出码 3，以及据此对旧 3.4.1 系统代理状态作出的判断，均为测试夹具假阳性，不计入正式产品结论。

所有下表中的系统代理、恢复 journal 和网络结论，均来自普通 shell 启动的只读探针，或经已有 Windows Explorer 桌面 Shell 启动的客户端和安装器。Computer Use 只用于操作可见界面，不再用其直接启动进程后的 HKCU 视图判定系统代理。

## 网络一致性与异常退出时间线

正常连接后，direct、显式代理和系统代理三条网络路径的连通性均符合各自预期；断开后均恢复直连基线。探测记录只保存状态和耗时，不保存目标、出口或端口信息。

第一次强制结束核心：

```text
T+0          强制结束核心
T+约 0.674 s 系统代理关闭
T+约 1.150 s 新核心进程出现
T+约 2.693 s 重新连接完成
```

本轮死亡端点暴露窗口不超过约 0.674 s。只有新核心恢复后客户端才重新连接。连续第二次强制结束核心时，客户端停止继续拉起核心并保持安全断开，系统代理与 journal 均保持干净。

强制结束 Flutter 应用进程：

```text
T+0          强制结束应用
T+约 0.173 s 系统代理关闭
T+约 0.215 s 核心退出
下次启动     遗留 journal 清除
```

对 launcher 使用全树强杀则仍存在发布阻断：

```text
T+0          强制结束 launcher 全进程树
T+2 s        系统代理仍指向已经死亡的端点
重新启动客户端后约 1.660 s  系统代理关闭
重新启动客户端后约 2.177 s  journal 清除
```

最后一条不是通过项。它证明当前测试安装包中的进程树内 guardian 无法抵御整树终止。树外独立 guardian 源码和 native tests 已完成，但仍必须由最终安装包在真实进程树中复测。

随后把当前源码构建的 launcher 与已验证的 3.4.2 子程序组合成临时测试运行时，并通过普通 Explorer/shell 启动。真实连接后观测到两个 launcher、一个 Flutter 应用和一个核心；系统代理、原生恢复记录、JSON journal、核心监听和联网均就绪。应用的父进程是外层 launcher，独立 guardian 位于该主树之外。对外层 launcher 执行全树强杀后：

```text
T+约 0.990 s  系统代理关闭、原生恢复记录清除
T+约 0.993 s  应用、核心、外层 launcher 与独立 guardian 全部退出
下次启动约 1.212 s  清除旧子程序遗留的 JSON journal
```

这项动态测试证明树外 guardian 设计在宿主机真实代理状态下有效，但临时运行时仍复用了旧 Flutter 子程序，不能替代包含全部 Dart、RunOnce 与安装器修复的最终 CI 安装包，也不能替代 Home 25H2 虚拟机结果。

在同一临时运行时的真实连接中，单独强制结束树外 guardian 后，外层 launcher 在约 0.384 s 内启动了新的 guardian；新进程仍以同会话 Explorer 为父进程。整个重建窗口内系统代理没有掉线，Flutter 应用、核心进程和联网探测均保持正常。该结果只证明源码 launcher 的 guardian 监督与重建路径，最终 CI 安装包仍需重复本项，并继续执行“重建后再强杀主树”的组合故障。

## 已复现问题与源码修复

### W25H2-01：代理恢复记录删除竞态

状态：**3.4.1 源码风险 / 旧 sky 实机归因作废 / 3.4.2 恢复路径实机 PASS**

launcher 已恢复代理并删除原生注册表记录，但 Dart JSON 记录仍存在时，应用重启会再次执行恢复。Windows PowerShell 5.1 在不存在的注册表路径上执行 `Remove-Item` 会返回非零状态，导致恢复永久停在 `recoveryPending`。同一缺陷也影响首次创建原生备份前的清理。

修复为在两处删除前使用 `Test-Path -LiteralPath`，只在路径存在时执行 `Remove-Item -LiteralPath`。代理与关闭恢复静态回归 12/12 通过，Windows PowerShell 5.1 兼容检查 7/7 通过。3.4.2 经 Explorer 启动后，应用强杀场景可在下次启动清除遗留 journal。此前经 `sky.launch_app` 得出的旧 3.4.1 代理判断受虚拟 HKCU 污染，不再作为该缺陷的实机证据。

### W25H2-02：正常退出时 Flutter 子窗口销毁重入

状态：**3.4.1 FAIL / 3.4.2 实机 PASS**

3.4.1 连续五次正常退出均在 Flutter 子 HWND 处理父窗口销毁通知时发生 `WM_PARENTNOTIFY` / `WM_DESTROY` 重入，并从空 view 对象偏移读取。修复后的顺序为：消息循环结束、显式 `window.Destroy()`、`CoUninitialize()`、标记正常关闭。3.4.2 连续 5/5 次优雅退出无新增 crash dump。

### W25H2-03：安装器提前启动客户端

状态：**3.4.1 FAIL / 3.4.2 安装脚本已修复 / 最终新版 PENDING**

启动项改为完成页的可选“启动 SSRVPN”，并加入 `postinstall` 和 `skipifsilent`。静默安装不再隐式启动 GUI，安装包 smoke test 会在安装后显式启动 launcher。

### W25H2-04：WebView2 缓存未清理

状态：**3.4.1 FAIL / 3.4.2 安装与卸载实机 PASS**

升级安装和卸载会精确清理两个已知 `EBWebView` 子树，不删除其父目录。3.4.1 覆盖升级、连接中卸载和保留数据后的静默重装均确认缓存已清理；安装器配置回归 18/18 通过。

### W25H2-05：在线升级确定性清空订阅与设置

状态：**3.4.1 已知设计 / 3.4.2 覆盖升级实机 PASS**

3.3.3 起的安装器会同时删除安装目录内数据和 LocalAppData 回退数据。因此旧逻辑下的内置更新即使安装成功，也会确定性清空订阅、设置和 DPAPI 密钥。3.4.2 改为只替换安装目录根文件、`bin` 顶层程序文件和 Flutter 运行时目录，明确保留安装版数据、LocalAppData 回退数据和窗口状态。3.4.1 到 3.4.2 覆盖升级前后的持久数据文件哈希一致；卸载后保留数据再静默重装，订阅刷新仍得到 25 个节点。

### W25H2-06：PowerShell 更新兜底可能随旧进程一起被终止

状态：**3.4.1 风险确认 / Explorer 安装交接实机 PASS / 内置更新仍 BLOCKED**

Flutter 应用属于外层 launcher 的 Job Object。旧 PowerShell fallback 及其启动的安装器没有可验证的 breakaway 标志；安装器关闭旧应用后，launcher 的 `TerminateJobObject` 可能把安装器一起终止。3.4.2 先用 `GetShellWindow` 确认已有桌面 Shell，再通过现有 Windows Explorer shell 交接安装包；Shell 不存在或交接失败时显示已校验文件路径，要求退出后手动运行，不再从受约束的应用进程创建兜底安装链。Explorer 启动的 3.4.1 覆盖升级和运行中同版本重装均已通过；客户端内置更新仍需最终发布资产。

### W25H2-07：代理未恢复仍停止核心

状态：**3.4.1 风险确认 / 3.4.2 普通 shell 故障注入 PASS**

旧安装脚本在精确恢复和禁用自有端点都失败后仍继续停止应用与核心，并可能返回成功。3.4.2 在任何 `Stop-Process` 前重新读取 Internet Settings：仍指向恢复记录中的端点、仍符合 SSRVPN 指纹、精确安装路径进程仍运行且系统代理指向本地端点，或注册表无法确认时，以非零状态阻断安装和卸载。普通 shell 中三轮故障注入均逐字段恢复测试前代理基线；不安全状态在停止进程前阻断，有效恢复记录则完成清理并保留用户原代理。经 `sky.launch_app` 启动安装器产生的独立退出码 3 属于隔离 HKCU 假阳性，不包含在此 PASS 中。

### W25H2-08：Windows 规范化 AutoDetect=0 后误走部分恢复

状态：**实机观察 / 3.4.2 源码已修复 / 最终新版 PENDING**

### W25H2-15：代理已恢复但 journal 仍可重放

状态：**源码已修复且跨实现回归 PASS / 最终安装包 live PENDING**

### W25H2-17：guardian 重建失败后的恢复与终止不一致

状态：**源码已修复且 native 构建 PASS / 最终安装包故障注入 PENDING**

旧启动链可能在 guardian 尚未 commit 时恢复被挂起的应用，或在 commit 后只由外层 launcher 恢复线程；前者会让应用在无保护状态运行，后者会在外层 launcher 突然退出时留下永久挂起进程。源码现要求 ready/alive → commit → post-commit alive → resume 的完整握手，guardian 与外层 launcher 都只做幂等恢复；没有 commit 绝不恢复应用。初始 Job、guardian、ready、commit 或存活校验任一步失败，均先恢复代理并保持子进程挂起，随后 fail-closed 清理。

guardian 重建失败后的恢复、可见安全降级、受保护进程终止、命名 Job 终止和 `KILL_ON_JOB_CLOSE` 后置确认现全部放在同一个跨进程代理事务锁内，避免 Dart 在恢复与强杀之间重新接管代理。若代理已确认安全而进程仍无法结束，会把残留窗口改为安全断开标题并显示置顶错误，禁止静默直连。最终安装包仍须用可控故障注入验证启动失败、commit 边界、重建失败及进程后置条件。
