# SSRVPN 三端深度可靠性审查与修复报告

日期：2026-07-22（Asia/Shanghai）
基线：`origin/main` v3.4.11（`3939405`）
工作分支：`audit/deep-reliability-hardening`
范围：Android、macOS、Windows；仅代码、自动化与本机非破坏性验证，不包含真实设备验收。

## 结论

本轮确认并修复了连接生命周期、端口竞争、系统代理/TUN 所有权、更新文件发布、Windows 覆盖安装、响应式 UI、订阅节点与规则合成中的多类竞态和失效路径。随后逐项核查用户补充的 9 个问题，其中第 1、5 项真实并已修复，其余 7 项已有实现或保护，未做重复修改；又针对 Windows 11 Build 26200 用户反馈，修复了 GUI 对隐藏/暂未完整可见 TUN 网卡的误阻断。最终代码审查未发现未解决的 Critical/Required 问题，仓库级 `make verify` 全部通过，代码进入 GitHub 合并与发版流程。

## 1. 连接意外断开、保活与退出

- 将“用户仍希望连接”与“当前核心是否健康”分离；自动恢复绑定连接代次，三端最多允许两次有界恢复并显示 `1/2`、`2/2`，用户断开、退出或新连接操作始终优先。
- 恢复前重新生成运行配置并重新分配端口；仅在明确的 bind 冲突时重试一次，不按端口号杀进程，也不接管其他软件的监听端口。
- IPv4/IPv6 回环同时探测，端口候选次数有上限；启动时再次确认实际绑定，封闭预检到启动之间的竞态。
- macOS TUN 按核心、授权服务、请求代次、DNS 日志与配置共同判断状态；网络服务被确认删除时安全退休旧 DNS 恢复状态，不把瞬时命令失败误判为已删除。
- Windows 核心记录升级为版本化的 PID、创建时间、会话和规范可执行路径；PID 复用、旧记录、跨安装路径和身份读取失败均拒绝终止未知进程。
- Android 对原生服务的终止态进行短周期有限重试和低频协调；新连接代次可取消旧回调，避免 Flutter 状态与原生 VPN 状态分叉。
- 系统代理使用三态所有权（持有、外部修改、不可确认）。健康恢复或异常退出清理期间若代理被用户/其他软件改动，会取消自动重连且不重新抢占。
- 正常退出、强制退出恢复、清理失败、TUN 残留和核心仍存活均有可见状态；只有端点已安全释放时 UI 才转为已断开。

## 2. UI 自适应与长文字

- Android、macOS、Windows 的教程、启动提示、运行提示、诊断页头和更新错误改为受视口约束的可滚动布局。
- 长节点名、订阅名和文件路径保留完整值，使用弹性布局、Tooltip/悬停或按需查看；没有用截断值参与配置或节点选择。
- 保留节点卡片原有点击、长按和键盘操作，不以 Tooltip 覆盖手势。
- 已覆盖紧凑窗口、最大文字缩放、堆叠通知和超长错误文本的 widget 测试。

## 3. 更新与覆盖安装

- OSS 仍为首选更新源，GitHub 为备用；下载恢复只接受元数据给出的预期 SHA-256，大小、扫描数量、哈希总量和锁等待均有限制。
- `.previous`/中断文件先复制到私有暂存并重新哈希，不再凭文件名或旧状态直接提升为正式安装包。
- 正式文件使用“目标不存在才成功”的原子硬链接发布；并发下载、其他进程或用户文件不会被 rename 覆盖。发布后再次校验哈希和长度。
- macOS 更新文件名带版本，避免多个版本共用模糊目标。
- macOS 安装包通过校验后，先调用统一连接事务安全断开；若核心无法停止则阻止打开 DMG。更新对话框同时明确要求覆盖旧应用前彻底退出当前 SSRVPN，避免运行中替换。
- Windows 覆盖安装使用同卷事务目录、精确清单和恢复日志；程序文件失败可回滚到上一可运行版本，用户数据 `bin\ssrvpn` 不进入替换或回滚范围。
- Windows 恢复日志执行严格 schema、路径、深度、文件/目录数量、单文件与总字节限制；破损、伪造或过大的日志在任何破坏性遍历前失败关闭。

## 4. 订阅、节点、规则与配置

- 订阅只导入可运行代理节点；运行时内建组名统一保留并做 Unicode/大小写规范化碰撞检查。
- 编辑节点、合并订阅和最终配置使用同一名称策略；歧义或重复名称直接报错，不生成 Mihomo 难以解释的配置。
- 保留合法 YAML 数据标量与空白，只移除非法控制字符；无效 YAML 失败关闭。
- 强制代理网站规则确定性去重并位于 DIRECT 之前；内建规则、额外组和代理引用在最终解析后检查唯一性和引用有效性。
- API secret 的配置写入与 API Authorization 使用同一规范化策略。

## 10. Windows 11 Build 26200 / TUN 专项核查

### 核查结论

- 用户反馈的核心现象成立：Mihomo 手动运行可正常启动 TUN 并转发流量时，SSRVPN 不应因为 Windows 额外网卡/路由探针暂时看不到完整状态而主动判定连接失败。
- SSRVPN 源码中不存在对 `SWD`、`PHANTOM`、PnP 设备状态或 `WIN32_EXIT_CODE: 31` 的直接拦截；实际误阻断来自 GUI 生命周期要求探针同时观察到预期双栈地址和分段路由，以及将已记忆但无地址、无路由的隐藏网卡空壳视为残留。
- “Mihomo 使用 gVisor，所以完全不需要 Wintun”不成立。Windows 上 gVisor 是用户态网络栈，但 sing-tun 仍通过 Wintun 设备收发数据。本轮没有删除管理员权限、Wintun/TUN 设备、核心进程、Mihomo API 或 `tun.enable` 的真实依赖。
- 截至核查时，Wintun 官方仍发布 0.14.1，没有可直接替换的官方更新版；因此无法把 Build 26200 的现象可靠归因并通过“升级 Wintun”解决。结合用户手动运行同一类核心成功的证据，本轮修复 GUI 假阴性，而不臆造上游驱动升级。

### 修复内容

- Mihomo API 可访问且运行配置 `tun.enable` 为真时，Windows 网卡地址/路由探针改为诊断告警，不再触发健康恢复或断开；相同状态只记录一次，恢复可见时再记录恢复日志。
- TUN 已启动但 Windows 暂时未暴露可持久化网卡身份时，保留启动前基线并继续连接，不再清理一个已工作的核心。
- 已记忆网卡若同时没有地址和路由，视为不会黑洞流量的隐藏空壳，不阻止重连；只要仍存在实际地址或路由，残留检查继续失败关闭。
- 管理员权限、Mihomo API、实际 `tun.enable`、核心身份、启动守护、端口所有权以及真实地址/路由清理仍是硬性安全边界。

### 新增验证

- Windows TUN 运行健康测试覆盖网卡缺失、路由不完整、探针异常和恢复可见，均验证不会误杀 API 与 TUN listener 已就绪的核心。
- Windows TUN 残留测试覆盖隐藏空壳、真实地址残留和真实路由残留，确认只放宽空壳场景。
- 生命周期静态门禁确认网卡身份暂时无法持久化时不会返回失败，并继续约束启动提交顺序。
